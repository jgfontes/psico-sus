# PsicoSUS Postman Collection

Covers all 25 REST endpoints across the 6 microservices: `auth`, `availability`,
`supervision`, `queue`, `session`, `medical-record`.

## Setup

1. `docker compose up` from the repo root (or run each service locally on its
   default port: auth 8080, queue 8081, availability 8082, session 8083,
   supervision 8084, medical-record 8085).
2. In Postman, import both files from this folder:
   - `psicosus.postman_collection.json`
   - `psicosus.postman_environment.json`
3. Select the **PsicoSUS - Local (Docker Compose)** environment.
4. If your stack overrides `JWT_SECRET`, update the `jwtSecret` environment
   variable to match — a couple of requests mint their own token locally and
   need the same secret the services are signing/validating with (see below).

## How chaining works

Requests capture IDs and tokens from responses into **collection variables**
via each request's `Tests` tab (e.g. `studentId`, `studentToken`,
`sessionId`...). As long as you run requests roughly in the order below, later
requests will already have what they need filled in — no manual copy-pasting.

CPFs/CNPJs/emails that must be unique are auto-generated by each request's
`Pre-request Script` the first time it runs (`ensureVar(...)`), so reruns
don't collide on unique constraints.

## Notes / gotchas carried over from the code review

- **`/auth/register` is public and doesn't verify `referenceId`.** The
  collection follows the intended flow (create the entity first, then
  register credentials against its id), but nothing in the API enforces that
  order or ownership — see the review notes from earlier in this session.
- **"internal" requests use a forged SERVICE token.** `POST /session/start`,
  `GET /availability/student/next`, and `GET /supervision/supervisor/by-crp/{crp}`
  are annotated `hasRole('SERVICE')` and are normally only ever called
  machine-to-machine — there's no public endpoint that issues a SERVICE token.
  A shared pre-request script (`mintServiceToken()`, defined at the collection
  level) builds one locally: HS256, signed with `SHA-256(jwtSecret)`, exactly
  matching `JwtKeyProvider`/`InternalTokenService` in the Java code. This only
  works if `jwtSecret` in your environment matches the real `JWT_SECRET` the
  stack is running with.
- **`Validate Internship Hours` needs a manual `internshipHoursId`.** The
  summary endpoint (`GET .../internship-hours/student/{studentId}`) only
  returns `sessionId` per record, not the row's own id, so it can't be
  auto-captured — pull it from the DB (`medical_record.internship_hours`) if
  you want to exercise that request.
